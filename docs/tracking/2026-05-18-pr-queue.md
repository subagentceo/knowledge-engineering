## State definitions

- **BLOCKED**: PR is mergeable but required CI checks haven't completed SUCCESS on its head SHA. Most common cause after OAUTO12+OAUTO14: PR head SHA was created before the workflow fixes, so its `osv-scanner.yml` produces the wrong context name. Fix: `gh pr update-branch <N>` from alex-jadecli (human actor) → fires `pull_request: synchronize` → CI runs naturally → gate satisfied.
- **DIRTY**: Real merge conflict with main. Requires manual checkout + rebase + resolve.
- **BEHIND**: Branch is behind main but not in conflict yet. `gh pr update-branch` fixes it.
- **draft**: Excluded from auto-merge by GitHub; needs `gh pr ready <N>` from operator first.

## How to drain each treatment

### Mechanical (BLOCKED, ready, automerge)

```bash
/tmp/merge-loop.sh <PR_NUMBER>
```

`merge-loop.sh` does: `gh pr update-branch` → watches for MERGED or required-check FAILURE → exits with status.

## Root cause discoveries this session

1. **OAUTO12 + OAUTO14 chain** — auto-rebase via `GITHUB_TOKEN` triggers GitHub anti-recursion (no downstream workflow firing). OAUTO12 added explicit `gh workflow run` redispatch; OAUTO14 fixed osv-scanner's `scan-pr` job to also run on `workflow_dispatch` so the redispatched run produces the ruleset-required context name `OSV-Scanner (PR) / osv-scan`.

2. **The actual unblock mechanism**: `gh pr update-branch <N>` from a human actor (alex-jadecli) — not from `GITHUB_TOKEN`. This fires `pull_request: synchronize` which doesn't trigger anti-recursion. The OAUTO14 workflow fix becomes important when a future cron-driven recovery runs on a stuck PR.

3. **`third_party/` is NOT in CI** — operator hypothesized it was the source of CI slowness; investigation showed it's only locally present (never tracked) so CI is unaffected. But locally, 13,689 files inflate Grep/Glob/find calls. OHYG1 codifies it in gitignore + adds OSV `--experimental-exclude` belt-and-suspenders.
