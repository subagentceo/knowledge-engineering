## State definitions

- **BEHIND**: Branch is behind main. `gh pr update-branch <N>` from **alex-jadecli (human actor)** fires `pull_request: synchronize` → CI runs naturally → gate satisfied → PR merges via auto-merge label. This is the unblock mechanism — NOT `gh workflow run` from `GITHUB_TOKEN` (which triggers anti-recursion suppression).
- **DIRTY**: Real merge conflict with main. Requires manual checkout + rebase + resolve + push.
- **draft**: Excluded from auto-merge by GitHub. Operator must `gh pr ready <N>` before the loop can act.

## Root cause discoveries this session

1. **OAUTO12 + OAUTO14 chain** — auto-rebase via `GITHUB_TOKEN` triggers GitHub anti-recursion (no downstream workflow firing). OAUTO12 added explicit `gh workflow run` redispatch; OAUTO14 fixed osv-scanner's `scan-pr` job to also run on `workflow_dispatch` so the redispatched run produces the ruleset-required context `OSV-Scanner (PR) / osv-scan`.
2. **Actual unblock mechanism**: `gh pr update-branch <N>` from a human actor (alex-jadecli) — not from `GITHUB_TOKEN`. Fires `pull_request: synchronize` which doesn't trigger anti-recursion. OAUTO14 workflow fix is the safety net for future cron-driven recovery.
