Provider-Agnostic Inference Gateway + Capture

One gateway, any OpenAI-compatible backend, uniform capture into your warehouse — on Cloudflare.
Proposal v2 · for your review

The shift since v1: the gateway shouldn't be Anthropic-specific. The OpenAI /v1/chat/completions format is the interop standard — verified that Anthropic exposes it at api.anthropic.com/v1/, and it's the same wire format Docker Model Runner, Cloudflare Workers AI, Vertex, Azure, and Bedrock all speak. So the gateway normalizes on that one format, fronts any of them, and logs every turn the same way. That's also what frees you from an Anthropic API key: point it at Workers AI (your CF account) or a local Docker Model Runner instead.

Architecture

the thing I build your captured data on-device (separate lane) existing / external

Two capture surfaces — kept honestly separate

A · HTTP gateway (this Worker)

Fronts any OpenAI-compatible provider. Captures Claude Desktop (via Configure Third-Party Inference) and any app/SDK pointed at the gateway URL. This is the buildable core.

B · On-device SDK (not gateway)

Apple Foundation Models & Chrome Gemma/Nano run on-device — no HTTP for a gateway to see. Capture in-app at the SDK level if wanted. Relevant to the coworkers Swift app: it can swap Apple on-device ↔ Claude per session.

Build plan — incremental, each slice ships on its own

1

Transparent proxy

Worker forwards one chat/completions request to a provider you pick (start: Cloudflare Workers AI) and streams it back byte-identical. Point Claude Desktop at it; confirm a chat round-trips. No logging yet.

2

Normalize + log to D1

Capture the normalized request/response + usage tokens into a raw capture_log on D1. Provider-agnostic because the schema is the OpenAI format.

3

Map into claude_ai.*

Parse the log into chat / chat_message / chat_message_content with deterministic PK upserts (idempotent — your composite keys).

4

Multi-provider + read surface

Provider switch (Workers AI / Docker Model Runner / Vertex…), then a small read endpoint — the seed of the coworkers app, prototyped on Workers first.

Honest open questions — I will not assume these

must verify before relying on it
  • Subscription vs. key: "Credential kind" has no fallback. Whether the gateway can forward your Max subscription to real Claude — vs. needing an API key, vs. a non-Claude provider — is unverified. Slice 1 tests it. (Provider-agnostic design sidesteps it: route to Workers AI / local instead.)
  • On-device ≠ gateway: Apple FM and Chrome Gemma/Nano bypass any HTTP gateway. I have not verified Chrome's exact built-in-AI capture API — that's a separate spike, not a claim.
  • Compat fidelity: Anthropic's OpenAI layer is "test, not production" and silently ignores fields. Capture is fine; don't route production Claude through it.
  • Scope: gateway captures go-forward Claude Desktop / app traffic — not historical claude.ai web/mobile chats (those still need the Settings → Export download).
