# Board Diagnosis

Structural audit of a CRM board — shaped as a CSM / admin deliverable. Surfaces missing-data patterns, stalled groups, column drift, automation gaps, and data-quality scores. Publishes a prioritized diagnosis doc in the user's workspace.

Flow: **Trigger → Gather → Synthesize → Publish (α, default) → Share (β, opt-in) → Proactive extension (opt-in, fix tasks)**.

## Input
- Optional: board name / ID via argument. Accepts `'all'` for multi-board audit (one doc per board, max 5 without explicit confirm).
- Optional: pre-declared mode (Default / Silent / Proactive).

## Output
- **α (default):** A `create_doc` diagnosis report in the user's workspace, title `Board Diagnosis — <board name> — <Mon DD>`. Body carries `Generated by Claude · <ISO timestamp>` + `<!-- claude-skill-id: board-diagnosis -->` for idempotency + auditability.
- **β (Share):** `create_notification` to CSM with doc URL.
- **Proactive extension:** `create_item` (fix-task) per issue, written into a `Fix Tasks` group with `Source = Claude` populated on a status column (auto-create the column with user confirm if missing); normalize trivial column data (phone, country code, email casing) via `change_item_column_values`; batch-edit owner / source / last-touch / stage columns when user confirms a plan. Never edits amount columns, never deletes.

## Knowledge
- Diagnosis strands (§ Step 5).
- Scoring formula (§ Step 5).
- Shared artifact conventions (§ Shared patterns).

## Tools (MCP)
- `get_user_context`, `list_workspaces`, `search` — board resolution.
- `get_board_info`, `get_column_type_info` — schema + column types.
- `get_board_items_page` — sampled pull (up to 2K most-recent).
- `board_insights` — aggregate diagnostics. **Note: has a known ~9% baseline error rate** — on failure, skip the strand and note it in the doc.
- `get_board_activity` — stalled-group + last-activity detection. Call as `get_board_activity(boardId)`. If the tool does not support a `days` parameter, derive staleness from item `updated_at` timestamps in the `get_board_items_page` results instead.
- `all_monday_api` — probe automation rules + doc-update mutations. No wrapped tool exists for either.
- `create_doc` — α publish.
- `create_item` — proactive fix-task creation.
- `change_item_column_values` — trivial normalization only (phone, country code, email casing).
- `create_notification`, `list_users_and_teams` — β share.

## Cross-skill handoffs
- **From morning-briefing:** If briefing flags >30% unowned deals or structural hygiene issues, route here.
- **From forecast-dashboard:** If >25% of deals lack forecast-critical fields, route here before building the dashboard.
- **To morning-briefing / forecast-dashboard:** Once the user fixes the flagged issues, they can re-run those skills.

## Step 0: Connector check

**Goal:** Fail fast if the user has no monday MCP connection. No partial writes, no guessing.

1. Try `mcp__monday__get_user_context`.
2. If the tool is missing, returns auth-style error, or the user has no accessible workspace:
   - Print exactly: *"I don't see the monday connector active on this session. Install it from https://monday.com/mcp, then run this skill again."*
   - Stop. Do not proceed.
3. If connector works, cache `user.id` and `user.name` for artifact metadata.

---

## Step 1: Detect mode

**Goal:** Default / Silent / Proactive.

- **Default** — batch confirm before any write; read-only audit by default.
- **Silent** — no intermediate chat output; runs through all steps and publishes doc without prompts.
- **Proactive** — ask once at session start:

> *"Proactive mode will create fix-task items, normalize trivial column data (phone, country, email casing), and may batch-edit owner / source / last-touch / stage columns where signal is clear. It will **not** edit amount columns, move items across workspaces, or delete anything. (a) Approve all, (b) Review each batch, (c) Deny all?"*

Cache the choice. Do not ask again during the session.

If undeclared, assume **Default**.

**Safety rail (canonical — referenced by Steps 1 and 8):** No deletes (boards, columns, items, groups). No amount-column writes. No cross-workspace moves. No writes outside a confirmed plan. Stage / owner / source / last-touch edits allowed inside a batched-confirm plan only.

---

## Step 2: Resolve the board (Gather)

**Goal:** Identify the target board(s) without ambiguity.

1. If argument provided → use it directly (board name or ID). If `'all'` → go to multi-board path below.
2. Else: check `get_user_context` favorites and `relevantBoards` for CRM-shaped boards (names containing deal, pipeline, lead, contact, opportunity, account, sales). If exactly one match → use it.
3. Else: `search("deal OR opportunity OR pipeline OR leads OR contacts")` across workspaces.
4. Else: `list_workspaces` → enumerate boards → `AskUserQuestion` with board names.

**Multi-board path ("all"):** Collect all resolved CRM boards. If count > 5, surface: *"Found N boards. Running all will take a while — want to pick a subset or continue with all N?"* Cap unconfirmed "all" runs at 5 boards. Run Steps 3–9 once per board, produce one doc per board.

---

## Step 3: Read board schema (Gather)

**Goal:** Full column inventory + type classification for every column (not just CRM-critical ones — the audit looks at ALL columns to find drift).

1. `get_board_info(boardId)` → full column list.
2. `get_column_type_info` for any column where fill rate will be calculated.

---

## Step 4: Pull data (Gather)

**Goal:** Enough data to compute fill-rate, staleness, and activity patterns. Paginate up to 2K most-recent items. Also pull:

- `board_insights` for aggregate stats. **Known ~9% baseline error rate** — if this call fails, skip strand 1 partial aggregate stats and note in doc; continue with item-level data.
- `get_board_activity(boardId)` for staleness detection. If the tool doesn't support a `days` param, derive last-activity from `updated_at` in item results — compute days since most recent update per group.
- `all_monday_api` GraphQL query for automation rules:
  ```graphql
  query { boards(ids: [<boardId>]) { automations { id name trigger { type } actions { type } } } }
  ```
  If this scope is unavailable, skip strand 4 with note: *"Couldn't inspect automations — ask admin to re-run."*

---

## Step 5: Synthesize — five diagnosis strands (Synthesize)

**Goal:** Produce a prioritized fix list across these strands:

1. **Missing data patterns** — required fields (stage, owner, value, close_date) blank on >25% of items. Compute fill-rate per column.
2. **Stalled groups** — groups with no item updates in >30 days.
3. **Column drift** — columns with data on <5% of items over last 90 days (likely abandoned). Suggest archive.
4. **Automation gaps** — stage-based automations absent on deal boards.
5. **Data-quality score** — composite 0-100 across all strands.

**Scoring formula:**

Each strand produces a strand score (0–100):
- **Missing data:** 100 − (15 × P0_count + 8 × P1_count + 3 × P2_count), floor 0
- **Stalled groups:** 100 − 15 per group stalled >30d, floor 0
- **Column drift:** 100 − 8 per abandoned column, floor 0
- **Automations:** 100 if all stage transitions have automations; −20 per missing stage automation, floor 0
- **Structure:** 100 − 10 per structural issue (duplicate columns, no groups, missing required column type), floor 0

**Overall score = weighted average:**
Missing data 40% + Stalled 20% + Column drift 15% + Automations 15% + Structure 10%

Severity classification:
- **P0** = fill-rate <50% on required fields, or stall >90d, or no stage automation. Blocks forecast / reporting.
- **P1** = fill-rate 50–75%, stall 30–90d, column <5% populated. Hygiene.
- **P2** = structural suggestions, minor cleanup.

---

## Step 6: Publish α — diagnosis doc (Publish)

**Goal:** `create_doc` in user's workspace with the full audit.

**Healthy board shortcut:** If overall score ≥ 90 and zero P0/P1 findings → skip doc creation and print:
*"Your board looks healthy (score N/100). No issues requiring a doc. P2 suggestions: [list inline]."* Stop here.

Core content sections:
- **Score:** overall data-quality score (0-100) + 5-strand breakdown.
- **P0 fixes** — concrete, actionable (format: `[P0] <issue> → <fix>`)
- **P1 / P2 fixes** — batched.
- **Automation recommendations** — per stage-based pattern missing.
- **Column cleanup** — archive list.

**Idempotency:** search for same-day doc by title + `<!-- claude-skill-id: board-diagnosis -->` marker. If found, use `all_monday_api` to append new findings to the existing doc body (don't overwrite). If `all_monday_api` doc-update fails, create a new doc with suffix ` — v2`.

Non-admin fallback isn't applicable (docs don't require admin). But if Proactive mode is on AND user isn't admin, skip `create_item` fix-tasks with a note ("admin-only").

---

## Step 7: Share β — route to CSM (opt-in)

**Goal:** Notify the CSM with the doc link.

1. `list_users_and_teams` → user picks CSM.
2. `create_notification` to CSM with doc URL and one-line summary.

Note: doc-permission grants via `all_monday_api` are unreliable due to workspace role complexity. Skip the permission mutation — the notification URL gives access if the doc is in a shared workspace. If the user needs explicit permission granting, direct them to monday.com manually.

---

## Step 8: Proactive extension — create fix-tasks + normalize (opt-in)

**Goal:** Turn the diagnosis into action, but stay inside the safety rail (see Step 1 for canonical definition).

For each P0 / P1 fix:
- `create_item` in a `Fix Tasks` group with title `Fix: <issue>`, assigned to board owner (or CSM if admin). Populate `Source = Claude` on the board's source/status column (auto-create the column with user confirm if missing).
- For **trivial** normalization (phone-number reformatting, country-code standardization, email lowercase): `change_item_column_values`. Never write to amount columns.
- Stage/owner/source/last-touch edits inside a batched-confirm plan are allowed (see safety rail in Step 1). Call `bulk-data-hygiene` for board-wide passes — this skill stays focused on diagnosis-driven point fixes.
- If >20 fix-tasks: batch into one `Fix Tasks — <date>` doc list (with `<!-- claude-skill-id: board-diagnosis -->` in body) rather than creating 20 items.
- Never deletes columns, never archives groups, never removes automations.

Normalization rules: phone → E.164 format, country → ISO 3166-1 alpha-2, email → lowercase.

---

## Step 9: Close the loop

**Goal:** Summary line + doc URL.

Print: `Published Board Diagnosis — <board> — <date>. Score <N>/100. <P0> critical · <P1> recommended · <P2> optional. <N> fix-tasks created. Doc: <url | "see summary above">.`

*Critical = blocks reporting, recommended = hygiene, optional = cleanup.*

## Shared patterns

- **No title prefix.** Artifact discoverability is via the `Generated by Claude · <ISO timestamp>` footer and the embedded `<!-- claude-skill-id: board-diagnosis -->` HTML comment in doc bodies. For items the plugin creates, use `Source = Claude` on a status column (auto-create the column with user confirm if missing).
- **Idempotency before create.** Always search for same-day artifact (by title + skill-id comment) first. Append via `all_monday_api`, fallback to ` — v2` suffix.
- **Safety rail.** No deletes, no amount-column writes, no cross-workspace moves. Stage/owner/last-touch/source edits OK in a batched-confirm plan only. Canonical definition in Step 1.
- **Ask once per session**, not per item, when entering Proactive mode.
- **Type-based column resolution**, not name-based — global teams have non-English columns.

## Error handling reference

| Failure | Behavior |
|---|---|
| Connector not installed | Step 0 stops; print install link. |
| No CRM board | Step 2 stops; offer path to `/monday-crm:workspace-builder`. |
| Board >10K items | Cap at 2K most-recent, flag sampling. |
| MCP tool error (retryable) | Exponential backoff 3x; on third fail, skip that strand, surface partial report. |
| Same-day artifact exists | Append via `all_monday_api`; fallback to ` — v2` suffix. |
| 429 rate limit | Backoff 3x, then stop with retry note. |
| Non-admin publish perms | Degrade to print-only. |
| Non-English column names | Resolve by type via `get_column_type_info`. |
| `board_insights` error | Skip aggregate stats strand; continue with item-level data; note in doc. |
| Automation probe fails | Skip strand 4; note "couldn't inspect automations — ask admin to re-run". |
